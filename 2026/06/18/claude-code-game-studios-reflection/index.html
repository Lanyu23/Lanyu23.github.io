发现一个把 Claude Code 变成游戏工作室的项目

一句话：这个项目想做的事情很直接，把一次 Claude Code 会话变成一个游戏开发工作室，让 AI 不只是帮你写代码，而是按游戏项目的完整流程陪你往前推。

最近又发现一个挺有意思的项目：Donchitos/Claude-Code-Game-Studios。

它的介绍很猛：Turn a single Claude Code session into a full game development studio。

简单说，就是把 Claude Code 从“一个会写代码的 AI 助手”，包装成一个有组织结构的游戏开发团队。里面不是只有一个通用 agent，而是准备了 49 个专业 agents、73 个 skills、12 个 hooks、11 套规则和 41 个文档模板。

它覆盖的也不是单纯写代码。

游戏设计、系统设计、程序开发、美术规范、音频、叙事、QA、性能、发布、热修复、版本记录，这些环节它都试图纳入流程里。

所以这个项目更像是一个“Claude Code 游戏开发工作室模板”。

你还是那个最终拍板的人，但它会帮你把游戏开发拆成一个个更专业的流程，让不同角色去负责不同事情。

这个项目到底是干嘛的

我理解它的核心价值，是给 AI 游戏开发加组织结构。

普通 Vibe Coding 的问题是，你往往只有一个 AI。你说想做一个功能，它就开始写。写得快是快，但它不一定会像一个游戏团队那样考虑整体流程。

比如一个游戏想法出来以后，正常来说不应该马上开写代码。

你得先想玩法核心是什么，目标玩家是谁，机制怎么闭环，数值怎么设计，UI 怎么承载，资产怎么命名，代码架构怎么分层，测试怎么做，发布前要检查什么。

这些事情如果全靠一个通用 AI 自己想，很容易漏。

Claude Code Game Studios 的做法，是把这些责任拆给不同角色。

它有 creative-director、technical-director、producer 这种导演和制作人角色，也有 game-designer、lead-programmer、art-director、qa-lead、release-manager 这种部门负责人。下面还有 gameplay-programmer、engine-programmer、ui-programmer、level-designer、ux-designer、qa-tester、performance-analyst 等更细的执行角色。

它甚至还针对 Godot、Unity、Unreal 都准备了不同的引擎专家。

这就有点像你不是在跟一个 AI 聊天，而是在让一个小型 AI 游戏团队围着你的项目转。

它不是自动驾驶，而是流程托管

这个项目有一点我觉得挺重要：它不是让 AI 完全自动做游戏。

README 里说得很清楚，它是 collaborative, not autonomous。

也就是它会问问题、给方案、展示选项、做草稿，然后等你批准。用户还是决策者，不是把项目丢给 AI 后就不管了。

这点我在使用过程中也有类似感受。

我只需要审批，给出自己的想法，它会一步步引导我把想法落地。它会问你现在项目处于什么阶段，是完全没想法、只有模糊概念、已经有清晰设计，还是已有项目要接入。然后根据你的回答，带你走不同流程。

这种体验非常省心。

以前自己做游戏，很容易卡在“不知道下一步该干嘛”。尤其是一个人开发的时候，脑子里同时要装玩法、程序、美术、UI、测试、发布，很容易顾此失彼。

用了这种项目之后，它会把下一步拆出来。

你只需要不断确认方向、补充想法、审批方案，它就能继续推进。

我觉得最有用的地方

第一，它能把模糊想法拆成可执行流程。

很多游戏一开始只是一个念头，比如“我想做一个什么玩法”。普通 AI 可能会马上帮你写原型，但这个项目会先走 brainstorm、design-system、create-epics、create-stories 之类的流程。它会尽量先把设计讲清楚，再拆任务，再进入开发。

第二，它让每个环节都有对应角色。

游戏开发最麻烦的地方，就是它不是纯程序项目。玩法好不好玩，数值合不合理，UI 顺不顺手，资产命名乱不乱，测试有没有覆盖，发布前有没有检查，这些都很重要。

一个通用 AI 很容易只盯着当前代码。

但如果有 QA、UX、技术总监、制作人、发行负责人这些角色参与，至少它会提醒你从更多角度看问题。

第三，它有 hooks 和规则兜底。

它的 hooks 会在 commit、push、资产变更、会话开始和结束等节点做检查。规则也会按路径约束，比如 gameplay 代码要数据驱动，UI 不应该持有游戏状态，tests 目录要符合测试命名和覆盖要求。

这对 AI 特别重要。

因为 AI 很容易为了快速完成当前需求，把规则先放一边。加了这些流程后，至少能让它在动手时多一层提醒。

第四，它适合长流程项目。

游戏项目不是写完一个功能就结束。它有原型、设计、实现、测试、打磨、发布、更新这些阶段。这个模板提供了 release-checklist、launch-checklist、patch-notes、hotfix、playtest-report 等流程，说明它想覆盖的是从想法到上线的完整周期。

使用中的感受

目前我还在用这个项目开发中，最终结果还没有看到。

所以现在还不能说它真的能把游戏做到什么质量。

但就使用过程来说，它确实让我感觉省心很多。

我只需要提出自己的想法，然后不断审批、补充、选择方案。它会把流程往下带，而不是让我自己每一步都去想“接下来该叫 AI 做什么”。

这点很关键。

以前用 AI 写代码，经常是你要自己当制作人、策划、主程、QA，还要自己写 prompt 管理它。现在这个项目相当于把这些角色和流程提前写好了，你只要按它的引导走。

当然，它也不是没有缺点。

最大的缺点就是耗 token。

用了这个项目后，我平均一天大概要用 3 亿 token。这个数字放在正常账号上，听起来就有点夸张了。

还好我现在用的是第三方供应商，模型是 xiaomimimo-v2.5-pro，1M 上下文，目前还有挺多 token，用起来也不太心疼。要是换成按官方额度烧，估计会很肉疼。

它为什么耗 token，其实也能理解。

因为它不是单 agent 短平快写代码，而是用一套工作室流程在跑。多个角色、多个文档、多个检查点、多个阶段，质量感和省心感是上来了，但 token 消耗肯定也跟着上来。

我的期待

我现在最想看的，是最终做出来的游戏效果到底怎么样。

流程再漂亮，最后还是要落到游戏本身。

游戏好不好玩，代码好不好维护，项目结构有没有保持清楚，AI 有没有真的减少我踩坑，最终都得等成品出来才能判断。

但至少目前这个阶段，我觉得它已经解决了一个很现实的问题：一个人用 AI 做游戏时，最缺的不是“有没有人写代码”，而是“有没有人帮你把整个开发流程管起来”。

Claude Code Game Studios 做的就是这件事。

它不只是写代码工具，更像是一个流程框架。

你负责想法和决策，它负责拆流程、叫角色、出方案、做执行、提醒检查。

如果后面最终成品效果不错，那这类项目的价值就很大了。因为它意味着一个人也可以借助 AI，把原本需要团队协作的游戏开发流程跑起来。

总结

这个项目给我的感觉是：AI 游戏开发正在从“让 AI 帮我写代码”，慢慢变成“让 AI 帮我组织整个项目”。

这两者差别很大。

写代码只是其中一环。

真正难的是把想法变成设计，把设计拆成任务，把任务实现出来，再测试、打磨、发布，并且过程中不把项目改成一团乱。

Claude Code Game Studios 试图把这些环节都纳入一个可执行的工作室流程。

目前我还在观察最终效果，但使用过程已经能明显感觉到，它确实能把人从很多琐碎流程里解放出来。

代价就是 token 烧得很快。

但如果 token 不是问题，那它确实是一个很值得尝试的 AI 游戏开发项目模板。
